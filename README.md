A voice-driven system design interview simulator built to the standard of a FAANG principal engineer interview. You speak your answers. The system listens, teaches, probes, draws, and scores — then tells you exactly where you stand.

Built around the curriculum of *System Design Interview — An Insider's Guide* (Volumes 1 and 2).

---

## The Two Agents

Every session is run by two agents. They have distinct voices, distinct roles, and a defined handover protocol between them.

**Alex** is a Senior Staff Engineer. He owns the teaching phase. Before the interview begins, Alex walks you through the core system design concepts relevant to your specific problem — load balancers, database replication, caching tiers, sharding, and so on. He uses analogies, flags what interviewers test hardest, and runs a comprehension check before handing you to Jordan. If you struggle with a concept, he reteaches it. His bar is calibrated to Jordan's: he will not pass you through until you can articulate what Jordan will ask.

**Jordan** is a Principal Engineer. He owns the interview. He asks the opening question, listens to your answer, issues follow-up probes when you miss something critical, and holds the session to FAANG hire-bar standards. When a concept benefits from a diagram — load balancer topology, master/slave replication, stateless web tier — he can ask you to draw it on the whiteboard and will score it against a rubric.

## How a Session Works

### 1. Problem statement
You provide a problem: *"Design a hotel reservation system"*, *"Design YouTube"*, *"Design a rate limiter"*. The system uses this to select which concepts Alex will teach and which probes Jordan will use.

### 2. Teaching phase (Alex)
Alex teaches the concepts needed for your specific problem in order of complexity — from single server setup through to full distributed architecture. Each concept gets an explanation, a real-world analogy, and a probe warning (what interviewers actually ask). For drawing concepts (load balancer, database replication, stateless web tier, sharding, full architecture), Alex activates the whiteboard and can ask you to sketch the topology.

When you tell Alex you are ready, he runs a comprehension check. If you pass, you hand over to Jordan. If not, he identifies the gap, reteaches it with a fresh example, and asks again.

### 3. Interview phase (Jordan)
Jordan opens with a scene — a concrete engineering context for the problem — then asks his opening question. Your answers are assessed in real time against a defined minimum bar per stage. Jordan issues targeted follow-up probes when you miss something; he has a probe limit per stage to keep the session moving. For concepts where spatial understanding matters, he can request a diagram, gate his verdict on it, and score it against the curriculum rubric.

If you want to go back to Alex at any point, you can. When you return, Jordan resumes from where he left off.

### 4. Debrief
At the end of the session you get a full breakdown: every stage, every verdict, concepts confirmed, concepts missed, and the internal notes Jordan made during assessment.

The curriculum is hard-coded from *System Design Interview — An Insider's Guide* and covers these 12 Chapter 1 concepts, taught in order:

| # | Concept | Drawing required |
|---|---------|-----------------|
| 1 | Single Server Setup | |

| 4 | Load Balancer | ✓ |
| 5 | Database Replication (Master/Slave) | ✓ |
| 6 | Cache Tier | ✓ |
| 7 | Content Delivery Network | |
| 8 | Stateless Web Tier | ✓ |
| 9 | Data Centers & GeoDNS | |
| 10 | Message Queue | |
| 11 | Database Sharding | ✓ |
| 12 | Full Architecture Capstone | ✓ |

Reference SVG diagrams are generated by Claude and cached globally. The whiteboard shows the reference diagram alongside your drawing for the six drawable concepts.

## Whiteboard

The whiteboard activates automatically when a drawing concept is reached. It shows:

- The reference architecture diagram (generated from curriculum)
- A drop zone for your uploaded drawing
- The rubric Jordan will score against (required items highlighted)
- A gate: if Jordan marked the diagram as required, you cannot submit your audio answer until you have uploaded a drawing

Jordan scores your drawing using Claude Vision against the curriculum rubric. Each rubric item gets PRESENT / PARTIAL / MISSING. A drawing that passes all required items can upgrade a PARTIAL verdict to CONFIRMED.

## Quick Start

```bash
# 1. Install UV (skip if already installed)
curl -LsSf https://astral.sh/uv/install.sh | sh

# 2. Clone and install
git clone https://github.com/martinhewing/connectionsphere-factory
cd connectionsphere-factory
uv sync

# 3. Configure
cp .env.example .env
# Edit .env — set the three required keys (see Configuration below)

# 4. Run
PYTHONPATH=src uv run uvicorn connectionsphere_factory.app:app --reload --port 8391
```

Open **http://localhost:8391/docs**

---

## Configuration

Three keys are required before the application will start.

```bash
# .env
ANTHROPIC_API_KEY=sk-ant-...    # console.anthropic.com
CARTESIA_API_KEY=...             # play.cartesia.ai
FACTORY_API_KEY=...              # generate locally — see below
```

Generate `FACTORY_API_KEY`:

```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

All other configuration has working defaults. See `.env.example` for the full reference.

---

## Diagram Cache

Reference diagrams are generated on first request and cached in memory (Redis in production). To warm all 12 diagrams on startup:

```bash
curl -X POST -H "X-API-Key: $FACTORY_API_KEY" http://localhost:8391/diagrams/pregenerate
```

To force regeneration of a single concept after a curriculum change:

```bash
curl -X POST -H "X-API-Key: $FACTORY_API_KEY" http://localhost:8391/concept/load_balancer/diagram/invalidate
```

## Development

**Run tests**

